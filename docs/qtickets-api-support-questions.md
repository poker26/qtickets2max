# Вопросы в поддержку Qtickets (REST API и сценарий из GigaChat)

Документация REST: [qtickets.help — REST API](https://qtickets.help/article/rest-api/).

## Подтвердите для вашего аккаунта организатора

1. **Создание заказа через API** — доступен ли метод `POST https://qtickets.ru/api/rest/v1/orders` (или актуальный хост из кабинета) для **программного** оформления заказа? В публичном оглавлении статьи по REST перечислены чтение заказов и операции с билетами; создание заказа может требовать отдельного согласования.
2. **Обязательные поля тела запроса** — точный контракт для `data`: `payment_id`, `event_id`, `baskets` (поля `show_id`, `seat_id` / `offer_id`, `quantity`), объект `site` / `site.host`, валюта, кастомные поля для счёта и т.д.
3. **Ответ после создания** — подтвердите, что в ответе возвращается **`payment_url`** (или аналог) для перенаправления покупателя на оплату.
4. **Тестовый контур** — есть ли песочница или тестовый организатор для проверки без боевых платежей.
5. **Ограничения по местам** — для экскурсий без рассадки: как корректно заполнять `baskets` и нужен ли запрос `GET /shows/{show_id}/seats` с особыми параметрами.

## Уже поддерживаемое в репозитории

- Чтение заказа: `GET .../orders/{orderId}` — используется в [`src/qtickets-client.js`](../src/qtickets-client.js).
- Новый MCP-сервис [`gigachat-mcp`](../gigachat-mcp/) добавляет чтение мероприятий, сеансов и мест; создание заказа включается только флагом окружения (см. README пакета).

Ответы поддержки зафиксируйте в вашей внутренней wiki и при необходимости обновите примеры тел запросов в `gigachat-mcp`.
