# Поток оплаты: SberPay в GigaChat и Qtickets

## Два практических варианта

### 1. `redirect_widget` (рекомендуется на старте)

- **Идея:** MCP отдаёт пользователю **публичную ссылку на покупку** из данных мероприятия (`site_url` в ответе `GET /events/{id}`), без создания заказа через API.
- **Плюсы:** не нужен недокументированный или ограниченный `POST /orders`; меньше ответственности за платёжный контракт.
- **Минусы:** оплата происходит **вне** встроенного сценария SberPay в чате (если Сбер требует именно встроенную оплату — уточняйте в пилоте).
- **Настройка:** в [`gigachat-mcp`](../gigachat-mcp/) задайте `QTICKETS_CHECKOUT_MODE=redirect_widget` (значение по умолчанию).

### 2. `payment_url_after_create`

- **Идея:** бэкенд вызывает **`POST /orders`** в Qtickets (инструмент `qtickets_create_order`, только при `QTICKETS_ENABLE_CREATE_ORDER=true`), получает **`payment_url`**, передаёт пользователю для оплаты.
- **Плюсы:** заказ уже создан в Qtickets; единый номер заказа для поддержки и вебхуков.
- **Минусы:** нужно **официально подтвердить** у Qtickets тело запроса и доступность метода для вашего аккаунта (см. [qtickets-api-support-questions.md](./qtickets-api-support-questions.md)).
- **Связка с SberPay:** если пилот требует оплату картой внутри GigaChat, это отдельное согласование с Сбером — `payment_url` Qtickets может использоваться как редирект или не подойти; это решается на этапе пилота.

## Встроенный SberPay «без выхода из чата»

Требует **спецификации партнёра** от Сбера (пилот). Реализация на стороне этого репозитория — только подготовка данных о товаре и заказе (MCP + Qtickets API). Подключение именно SberPay к потоку оплаты — вне кода Qtickets, по инструкциям Сбера после допуска в программу.
